Módulo 1 — Banco de dados e histórico
Objetivo

Guardar histórico longo, preferencialmente desde 01/01/2000, ou desde a primeira data disponível em cada fonte.

Essa regra está correta:

Histórico desde 2000 como padrão mínimo desejado, ou desde a primeira data disponível quando a fonte tiver mais ou menos histórico.

Isso é obrigatório para:

tendência;
sazonalidade;
volatilidade;
correlação;
regressão;
previsão;
probabilidade;
machine learning;
economia aplicada.
Schemas fechados

CREATE SCHEMA IF NOT EXISTS raw;
CREATE SCHEMA IF NOT EXISTS staging;
CREATE SCHEMA IF NOT EXISTS dw;
CREATE SCHEMA IF NOT EXISTS ml;
CREATE SCHEMA IF NOT EXISTS app;

| Schema  | Função                                         |
| ------- | ---------------------------------------------- |
| raw     | dado bruto exatamente como veio da API/arquivo |
| staging | dado tratado e padronizado                     |
| dw      | modelo analítico final                         |
| ml      | bases de treino, features e previsões          |
| app     | tabelas prontas para o Streamlit               |


Tabela central histórica


CREATE TABLE IF NOT EXISTS dw.fato_serie_historica (
    id SERIAL PRIMARY KEY,
    data DATE NOT NULL,
    fonte TEXT NOT NULL,
    codigo_serie TEXT,
    indicador TEXT NOT NULL,
    categoria TEXT,
    subcategoria TEXT,
    pais TEXT DEFAULT 'Brasil',
    uf TEXT,
    municipio TEXT,
    regiao_ibge TEXT,
    regiao_comercial TEXT,
    valor NUMERIC,
    unidade TEXT,
    periodicidade TEXT,
    data_inicio_fonte DATE,
    data_fim_fonte DATE,
    data_coleta TIMESTAMP DEFAULT NOW(),
    hash_registro TEXT,
    UNIQUE (
        data,
        fonte,
        codigo_serie,
        indicador,
        pais,
        uf,
        municipio,
        regiao_ibge,
        regiao_comercial
    )
);

Controle de carga

CREATE TABLE IF NOT EXISTS app.etl_controle_carga (
    id SERIAL PRIMARY KEY,
    fonte TEXT NOT NULL,
    indicador TEXT NOT NULL,
    codigo_serie TEXT,
    data_inicio_solicitada DATE,
    data_inicio_disponivel DATE,
    data_fim_disponivel DATE,
    ultima_data_coletada DATE,
    status TEXT,
    mensagem TEXT,
    qtd_registros INT,
    data_execucao TIMESTAMP DEFAULT NOW()
);

Decisão fechada

Todo dado externo entra primeiro em raw, depois passa por staging, depois entra no dw.

Nada de jogar dado direto no dashboard.

Módulo 5 — Indicadores estatísticos

Aqui começa a parte analítica de verdade.

5.1 Variação mensal

var_mensal = (valor_mes_atual - valor_mes_anterior) / valor_mes_anterior
Uso:

venda;
volume;
preço;
dólar;
milho;
soja;
proteína;
importação.

5.2 Variação trimestral

var_tri = (media_ultimos_3_meses - media_3_meses_anteriores) / media_3_meses_anteriores

Uso:

tendência mais estável que a mensal;
redução de ruído;
análise de curto prazo.

5.3 Média móvel

media_movel_3m = média dos últimos 3 meses
media_movel_6m = média dos últimos 6 meses
media_movel_12m = média dos últimos 12 meses

Uso:

suavizar ruído;
identificar tendência;
comparar atual contra histórico.

5.4 Volatilidade
volatilidade = desvio_padrao(variações_mensais)

Uso:

saber se um indicador é instável;
medir risco;
compor o score de confiança.

5.5 Z-score

z = (valor_atual - média_histórica) / desvio_padrão_histórico

Uso:

detectar anomalias;
identificar alta ou queda fora do normal;
alertas.

Interpretação:

|      Z-score | Interpretação        |
| -----------: | -------------------- |
| menor que -2 | queda anormal        |
|       -2 a 2 | comportamento normal |
|  maior que 2 | alta anormal         |


5.6 Correlação

correlacao = corr(indicador_x, indicador_y)

Exemplos:

dólar x venda de salmão;
IPCA alimentos x volume vendido;
frango x tilápia;
milho x preço de proteína;
renda x mix premium.

(Uma Obs vamos ter uma de peixes vs outras proteínas tbm a pessoa seleciona o peixe vs a proteína)

Interpretação:

|    Correlação | Leitura                   |
| ------------: | ------------------------- |
| -1,00 a -0,70 | relação negativa forte    |
| -0,69 a -0,30 | relação negativa moderada |
|  -0,29 a 0,29 | relação fraca             |
|   0,30 a 0,69 | relação positiva moderada |
|   0,70 a 1,00 | relação positiva forte    |



5.7 Elasticidade

elasticidade = variação_percentual_volume / variação_percentual_preço

Uso:

sensibilidade a preço;
campanhas;
desconto;
troca de mix.

Exemplo:

Se o preço sobe 5% e o volume cai 10%:

elasticidade = -10% / 5% = -2

Interpretação:

| Elasticidade | Leitura                  |
| -----------: | ------------------------ |
| menor que -1 | produto sensível a preço |
| entre -1 e 0 | sensibilidade moderada   |
| próximo de 0 | pouco sensível           |

Módulo 7 — Cenário 1 a 10

Objetivo

Transformar vários indicadores em uma nota clara.

Fórmula base


score_final =
0.30 * oportunidade_regional +
0.20 * desempenho_vendas +
0.15 * competitividade_proteina +
0.15 * ambiente_economico +
0.10 * cobertura_comercial -
0.10 * risco queda

Depois converte para nota:

cenario_1_10 = ceil(score_final / 10)

Escala fechada

| Nota | Situação           | Ação                    |
| ---: | ------------------ | ----------------------- |
|    1 | crítico extremo    | ação urgente            |
|    2 | muito ruim         | plano corretivo         |
|    3 | ruim               | revisar preço/mix       |
|    4 | atenção            | acompanhar semanalmente |
|    5 | neutro             | manter                  |
|    6 | levemente positivo | testar campanha         |
|    7 | bom                | explorar oportunidade   |
|    8 | muito bom          | expandir comercialmente |
|    9 | excelente          | prioridade alta         |
|   10 | ideal              | máxima prioridade       |


Módulo 8 — Decisão vendedor/promotor/campanha
Regra inicial

| Condição                                             | Recomendação   |
| ---------------------------------------------------- | -------------- |
| oportunidade alta + poucos clientes                  | vendedor       |
| oportunidade alta + muitos clientes + baixa execução | promotor       |
| competitividade ruim + queda de volume               | campanha       |
| renda alta + mix premium baixo                       | ação premium   |
| risco alto + queda forte                             | ação corretiva |
| oportunidade baixa + risco alto                      | não expandir   |
| oportunidade média + risco baixo                     | monitorar      |


Fórmula de decisão

score_vendedor =
0.35 * oportunidade_regional +
0.25 * baixa_cobertura_clientes +
0.20 * potencial_consumo +
0.10 * crescimento_regional +
0.10 * margem_media

score_promotor =
0.30 * clientes_ativos +
0.25 * redes_ativas +
0.20 * baixa_execucao_loja +
0.15 * mix_premium +
0.10 * oportunidade campanha

score_campanha =
0.30 * risco_substituicao +
0.25 * queda_volume +
0.20 * competitividade_preco +
0.15 * sazonalidade +
0.10 * margem disponível

| Maior score      | Ação                       |
| ---------------- | -------------------------- |
| score_vendedor   | adicionar vendedor         |
| score_promotor   | adicionar promotor         |
| score_campanha   | campanha de marketing      |
| todos baixos     | monitorar                  |
| risco muito alto | corrigir antes de expandir |

O que está 100% decidido
O app começa no Brasil, não em MG.
Pode selecionar qualquer UF.
MG será o primeiro caso prático por causa do seu trabalho.
Histórico desde 2000 será padrão desejado.
Se a fonte tiver antes de 2000, salvamos também.
Se a fonte tiver só depois de 2000, salvamos desde onde existir.
Tudo será salvo no PostgreSQL.
O app separa dado real, estimado, previsto e recomendado.
As recomendações sempre terão confiança/probabilidade.
O app terá cenário de 1 a 10.
A primeira versão terá BCB + IBGE + vendas internas.
Depois entram CONAB, CEPEA, Comex e FRED/FMI.
